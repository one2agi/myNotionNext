2. 中国境内特定约束（高频被忽略）

⚠️ 备案（ICP）是"非对称硬阻断"

中国大陆腾讯云 VPS 在 80/443 端口解析到未备案域名时，运营商直接 RST。 用非标端口是短期绕过，但 2025 年端口扫描执法已收紧。 备案周期 7–20 个工作日，需要国内主体（个人备案已收紧）。

结论：方案 1（n8n on mainland VPS）= 备案阻断。改用腾讯云香港/海外节点可绕过备案， 但增加 > 80ms 延迟到 Z-Pay 回源，付出与方案 3 类似的代价却少一个 Free 配额优势。

🌐 出境到 notion.com 的稳定性

  • EdgeOne Pages：境内有 Tencent 自家边缘节点 + 走腾讯骨干网出境，相对最稳。
  • Cloudflare Workers：境内走 Anycast，节点在 HK/东京/新加坡，无合同 SLA 保障，理论上偶发丢包/超时。
  • n8n self-host on HK 节点：回源 Z-Pay（境外）≈ Workers；写 Notion 走境外，链路跟 Workers 接近。
  • Make SaaS：触发源域名 make.com 偶尔被 GFW 干扰，回调写 Notion 走 Make 出口，链路过两步。

结论：方案 4 在境内回源路径最短 + 自有骨干网，方案 3、n8n-on-HK、Make 都有"境外一跳"额外风险。

🚦 Notion REST 3 req/s 限速（4 方案共同的工程问题）

Notion API 文档明文：平均 3 req/s/连接，突发允许；429 = 限速，529 = 过载，两者均带 Retry-After 头。 POST /v1/pages 是非幂等——重试一次会创建重复行。

4 方案都必须实现：①读 Retry-After ②指数退避 + 抖动 ③以 Z-Pay trade_no 作为 natural key 幂等去重。 这是可行性门槛，不是差异化——方案 4 的 order-store.js 已经满足；切到任何方案都要重做一遍。

3. 推荐理由（3 条）

理由 ① — 沉没成本是真实成本

方案 4 是唯一已上线、被 E2E 测试覆盖、被安全审计过的方案。 切换到任意替代方案 = 至少 1–2 天迁移 + 重过安全审计 + 重新观察线上稳定性。 对 < 1000 单/月、单价 ¥79–¥299 的产品，迁移 ROI 是负数。

理由 ② — 隐性成本让"便宜方案"不便宜

方案 1 的真实成本 = VPS + 备案 + SSL + n8n 升级 + 监控 + 备份，> 5 项隐性支出，单价 ¥200+/月起。 方案 2 的真实成本 = 月费 + 流量 + Make 模板升级，€5–10/月起且无免费 tier。 方案 3 表面最便宜（0 元），但出境稳定性是隐性 SLA 风险。 方案 4 = 0 元 + Pages 自带 SSL + 腾讯骨干网 + 已审计 = 隐性成本几乎为 0。

理由 ③ — Notion API 升级背锅链最短

方案 1、2 依赖第三方维护者：n8n Notion 节点 / Make Notion module 升级滞后 2–8 周。 方案 3、4 写自己的 TypeScript：Notion SDK 升级时 yarn tsc 不通过就立刻知道。 项目当前的 TypeScript strict + CI 已经把这个回路走通，迁移 = 抛弃这条已经修好的回路。

5. 8 条高置信度 Findings（带 URL）

F1 · highCloudflare Workers Free 不按子请求计费，覆盖 Z-Pay 回调轻松。

Free 层 10 万 req/天、每次调用 10 ms CPU；fetch() 到 api.notion.com 是子请求，不加费。HMAC 验签亚毫秒，整次调用 << 10 ms，< 1000 单/月余量 > 99.9%。

Cloudflare 官方：Free tier = 100,000 requests/day and 10 ms CPU per invocation（原文）。Notion HTTPS POST 是子请求，所以 0 元增量。
来源：developers.cloudflare.com/workers/platform/pricing · …/limits

F2 · highEdgeOne Free Plan 没有合同 SLA——支付回调"可靠性期望"非合同化。

官方文档原文："Basic security and acceleration services are provided, but SLA is not guaranteed"；99.9% SLA 仅在付费档与独立 SLA 法律文档里出现。对支付回调意味着任何可用性期待都是 informal，不是 contract。

原文："SLA guarantee level will be adjusted from 99.9% to no guaranteed SLA."（Free Plan 文档）
来源：EdgeOne Free Plan Guide · EdgeOne SLA legal doc

F3 · high中国大陆腾讯云 VPS 公开域名必须备案；香港/海外节点豁免。

境内服务器 80/443 端口解析到未备案域名 → 运营商直接 RST。非标端口是短期绕过，2025 年端口扫描执法已收紧。香港/海外 VPS 完全不需要备案。

原文："中国香港地区及境外地区的云服务无需进行备案，也不能用于备案"。
来源：Tencent Cloud ICP Support · 境内备案 · HK 节点规则

F4 · highNotion REST 平均 3 req/s/连接，429/529 都带 Retry-After。

官方文档明文："an average of three requests per second per connection (bursts above average are allowed)"。429 = "This request exceeds the number of requests allowed. Slow down and try again."；529 = "Notion is temporarily overloaded. Respect the Retry-After response header and try again later."

推荐处理顺序：①先读 Retry-After；②用 queue 做后端退避；③exponential backoff + jitter（避免 thundering herd）。
来源：Notion Request Limits · Notion Status Codes

F5 · high4 方案都必须实现"幂等重试 + 指数退避 + 抖动"。

POST /v1/pages 非幂等——一次 429 重试就重复行。两种防御：①Stripe-style idempotency key；②natural-key dedup（用 Z-Pay trade_no，项目当前已经这样做）。这是可行性门槛，不是差异化。

AWS 工程文章警告：plain exponential backoff 会让多个客户端在同秒重试——必须加 jitter。
来源：Notion Request Limits · AWS: Exponential Backoff and Jitter · RFC 6585 (HTTP 429)

F6 · highNotion 的 webhook 是 at-most-once（8 次重试 / 24h，无日志/无手动重放）——反方向用 API 直写反而更稳。

原方向是 Notion → 消费者，不适用 Z-Pay → Notion；但它说明 Notion 整体 webhook 生态"消费方成熟、提供方薄弱"，所以在 Z-Pay → Notion 方向上，由接收端自己调 REST API（方案 3、4 风格）比走 Notion webhook 范式更可靠。

原文："We aim for at-most-once event delivery... retry delivery up to 8 times using an exponential backoff schedule."Hookdeck 补充："Browsable log: Not available" / "Manual retry: Not available"。
来源：Notion Webhook Delivery · Hookdeck: Notion Webhook Guide

F7 · highEdgeOne Pages Functions 提供两种 serverless：Edge（毫秒级冷启动，全球边缘）+ Cloud（百毫秒冷启动，Node/Python/Go）。

当前 notify.ts 跑在 Cloud Function。如果未来要降冷启动，不用换平台，改用 Pages Edge Function 即可（毫秒级冷启动 + 仍 0 元）。这是方案 4 的隐藏升级空间。

原文："Edge Functions: Runs on EdgeOne global edge nodes, provides ultra-low latency and millisecond-level cold startup ... Cloud Functions: Deployed on cloud servers, supports Node.js, Python, and Go multiple runtime environments."
来源：Pages Functions Overview · Release Notes

F8 · highCloudflare Workers 调试链是生产可用的（wrangler tail）。

Wrangler v3 提供 --status / --method / --search / --sampling-rate / --header / --ip-address 过滤 + JSON 输出，jq 即可 triage Z-Pay 回调。已知 caveat：filter 改动生效有 ~10s 延迟（cloudflare/workers-sdk#8863）。

示例：wrangler tail --format=json --status=error | jq 'select(.request.url|test("zpay"))' 可以即时定位 Z-Pay 回调 4xx/5xx。
来源：Wrangler v3 docs · Dashboard Logs blog
